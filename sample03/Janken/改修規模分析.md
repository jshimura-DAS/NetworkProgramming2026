## 現状の仕様

### アーキテクチャ
- **接続形態**: 1対1（クライアント vs ホスト）
- **対戦相手**: ホストがランダムな手を生成
- **通信フロー**: 同期的・単一セッション
- **接続数**: 1クライアントのみ

### 処理フロー
```
[Client] → 接続 → [Host]
[Client] → 手を送信 → [Host]
[Host] → ランダム手生成
[Host] → 勝敗判定
[Host] → 結果送信 → [Client]
接続終了
```

---

## 新仕様の概要

### 要求仕様
- **接続形態**: 1対多（ホスト + 複数クライアント）
- **対戦相手**: クライアント同士
- **マッチング**: ホストが2人のクライアントをペアリング
- **通信フロー**: 非同期・複数セッション管理

### 新しい処理フロー
```
[Client1] → 接続 → [Host]
[Client2] → 接続 → [Host]
[Host] → マッチング(Client1 vs Client2)
[Client1] → 手を送信 → [Host]
[Client2] → 手を送信 → [Host]
[Host] → 両方の手を受信待機
[Host] → 勝敗判定
[Host] → 結果送信 → [Client1]
[Host] → 結果送信 → [Client2]
次のマッチへ or 接続維持
```

## 詳細な改修内容

### 1. JankenHost.cs の改修（★★★★★ 大規模）

#### 1.1 必要な新機能

##### A. 接続管理機能
```
現状: 1つのAccept()で終了
新仕様: 継続的なAccept()ループ
```

**必要な実装:**
- 接続リスナーの継続実行
- 接続クライアントのリスト管理
- クライアントIDの割り当て
- 接続・切断の検知

**コード規模:** 約50-80行

##### B. クライアント状態管理

**必要なデータ構造:**
```csharp
public class ClientInfo
{
    public int ClientId { get; set; }
    public Socket Socket { get; set; }
    public string Name { get; set; }
    public ClientState State { get; set; }
    public Hand? SelectedHand { get; set; }
    public DateTime ConnectedTime { get; set; }
}

public enum ClientState
{
    Connected,      // 接続済み
    Waiting,        // 待機中
    Selecting,      // 手を選択中
    Ready,          // 準備完了
    InGame,         // 対戦中
    Disconnected    // 切断
}
```

**コード規模:** 約40-60行

##### C. マッチング機能

**必要な実装:**
- 待機中のクライアントをキューに追加
- 2人揃ったらマッチング成立
- マッチ情報の作成と管理
- マッチ解除処理

**マッチング戦略:**
- **先着順**: 接続順に2人ずつペアリング
- **ランダム**: ランダムにペアリング
- **ルーム制**: クライアントがルームを選択

**コード規模:** 約60-100行

##### D. 非同期通信処理

**技術スタック:**
```csharp
// 使用する.NETの機能
- Task / Task<T>
- async / await
- ConcurrentDictionary<TKey, TValue>
- ConcurrentQueue<T>
- SemaphoreSlim
- CancellationToken
```

**コード規模:** 約100-150行

##### E. ゲーム進行管理

**必要な実装:**
- ゲームセッションの作成
- 両プレイヤーの手の収集
- タイムアウト処理
- 結果の配信
- ゲーム終了処理

**コード規模:** 約80-120行

##### F. 新しいクラス構造（提案）


**新規コード規模:** 約250-400行

```csharp
public class Match
```

#### 1. マルチスレッド・同期処理（難易度: 高）

## まとめ

### 改修規模の結論

**改修規模: 大規模（★★★★☆）**

#### 理由:
1. **アーキテクチャの根本的変更**
   - 同期→非同期
   - 単一接続→複数接続
   - 1対1→多対多

2. **新規機能の追加**
   - クライアント管理
   - マッチング
   - 状態管理
   - ゲームセッション管理

3. **技術的複雑度の上昇**
   - マルチスレッド
   - 並行処理
   - 複雑なエラーハンドリング

4. **コード量の大幅増加**
   - 約3倍～4倍のコード量
   - 新しいクラス構造の導入

### 実現可能性

**実現可能性: 高（十分実現可能）**

ただし、以下の条件下で:
- 段階的な実装アプローチ
- 十分な開発期間（6-9週間）
- async/awaitの理解
- 適切なテスト

### 学習効果

このプロジェクトを通じて習得できる技術:
- ✓ ネットワークプログラミング（上級）
